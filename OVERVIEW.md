# Election Radar — 프로젝트 개요

## 1. 프로젝트의 의의

### 선거 보도의 정보 비대칭 해소

대한민국의 재보궐선거는 전국적 관심을 받는 대선·총선에 비해 언론 보도량이 적고, 유권자가 판세를 파악하기 어렵다. Election Radar는 분산된 뉴스 기사를 자동으로 수집·분석하여 **일반 시민이 선거 판세를 한눈에 파악할 수 있는 대시보드**를 제공한다.

### AI 기반 여론 분석의 새로운 접근

기존 여론조사는 표본 크기, 조사 시점, 응답 편향 등의 한계를 갖는다. Election Radar는 여론조사를 대체하는 것이 아니라, **뉴스 기사라는 비정형 텍스트 데이터를 RAG(Retrieval-Augmented Generation) 기술로 분석**하여 여론의 흐름과 맥락을 보완적으로 제공한다. LLM이 단순 감성 분석을 넘어 기사의 논조, 정책 평가, 후보 간 비교 등을 종합적으로 판단하여 우세/균형/열세를 판정한다.

### 기술적 의의

- **RAG 파이프라인의 실전 적용**: 학술 데모 수준이 아닌, 실시간 데이터 수집부터 사용자 대시보드까지 이어지는 완전한 End-to-End 시스템
- **검색-분석 항목 1:1 매칭**: 커스텀 쿼리 템플릿의 8가지 검색 축(지지율·공약 반응·강점·약점·이슈·지지율 추이·출마 여론·선거 전략)이 LLM의 9가지 reasoning 항목과 체계적으로 연동
- **한국어 NLP 특화**: 한국어 문장 분리(kss), 한국어 임베딩(KoSimCSE), 한국 뉴스 매체 크롤링 등 한국어 환경에 최적화
- **교체 가능한 아키텍처**: Strategy + Registry 패턴으로 임베딩 모델, Vector DB, LLM 등 핵심 컴포넌트를 설정 파일 변경만으로 전환 가능

## 2. Election Radar의 역할

### 2.1 뉴스 자동 수집 엔진

네이버 뉴스와 정치 전문 매체(오마이뉴스, 프레시안, 미디어오늘)에서 선거 관련 기사를 **자동으로 크롤링**한다. 스케줄러(APScheduler)가 하루 3회(07:00, 12:00, 18:00) 수집을 실행하며, URL 중복 방지 시스템으로 같은 기사의 재수집을 차단한다.

```
수집 → 태깅 → 청킹 → 임베딩 → VectorDB 저장
```

### 2.2 후보·선거구 자동 분류기

수집된 기사를 후보명·정당명·선거구 키워드를 기반으로 **자동 태깅**한다. 키워드 주변 ±50자 윈도우 내 선거 맥락 단어 존재 여부를 검증하여 동음이의어 오태깅을 방지한다(예: "조국"(정치인) vs "조국"(motherland)). 어떤 후보에 대한 기사인지, 어떤 선거구와 관련되는지를 정확하게 판별하여 이후 RAG 검색의 정확도를 높인다.

### 2.3 AI 판세 분석기

Vector DB에 저장된 최근 기사를 커스텀 쿼리 템플릿(`query_templates.json`)으로 검색(Retrieve)하고, bi-encoder 필터링 + Cross-encoder 정밀 재평가의 **2단계 재정렬(Rerank)**을 거친 뒤, LLM(GPT-4o 또는 Claude)이 **후보별 9가지 항목을 분석하여 우세/균형/열세를 판정하고 승리 확률을 산출**한다.

**후보별 9가지 분석 항목**:
지지율, 공약 반응, 강점, 약점, 이슈, 지지율 추이, 출마 여론, 선거 전략, 예측

검색 쿼리와 LLM 분석 항목이 1:1로 매칭되어, 수집된 기사에서 각 항목에 해당하는 근거를 체계적으로 추출한다. 판정 근거로 사용된 기사 목록과 항목별 논거를 함께 제공하여 결과의 투명성을 보장한다.

### 2.4 판세 변화 추적기

매 판정 결과를 시계열 데이터로 누적 저장하여 **시간에 따른 판세 변화를 추적**한다. 특정 이슈가 발생한 시점 전후로 후보별 승률이 어떻게 변했는지를 시각적으로 확인할 수 있다.

### 2.5 시각화 대시보드

일반 사용자를 위한 직관적인 웹 대시보드를 제공한다:
- **선거구 선택**: 탭 방식으로 관심 선거구를 전환
- **판정 카드**: 후보별 승리 확률, 판정(우세/균형/열세), 9가지 분석 항목(지지율·공약 반응·강점·약점·이슈·지지율 추이·출마 여론·선거 전략·예측) 상세 표시
- **승률 추이 차트**: Recharts 기반 시계열 그래프로 판세 흐름을 시각화
- **관리자 페이지**: VectorDB 상태 모니터링, 파이프라인 수동 실행, RAG 설정 조정

## 3. 프로젝트의 중요성

### 3.1 민주주의와 정보 접근성

민주주의의 핵심은 유권자의 **정보에 기반한 선택**이다. 재보궐선거처럼 관심도가 상대적으로 낮은 선거에서는 유권자가 판세를 파악할 수 있는 정보원이 부족하다. Election Radar는 기술을 통해 이 격차를 줄이고, 유권자가 더 나은 정보를 바탕으로 투표 참여를 결정할 수 있도록 돕는다.

### 3.2 투명한 분석 과정

기존 선거 예측 서비스는 분석 방법론이 불투명한 경우가 많다. Election Radar는:
- **근거 기사 목록**을 판정 결과와 함께 제시하여 검증 가능성을 확보
- **오픈소스 코드**로 분석 알고리즘 전체를 공개
- **설정 파일 기반** 운영으로 파라미터 조정의 투명성 보장

이를 통해 "왜 이런 판정이 나왔는가?"를 사용자가 직접 추적하고 검증할 수 있다.

### 3.3 실시간성과 연속성

여론조사는 조사 시점의 스냅샷이며 주 1~2회 발표가 일반적이다. Election Radar는 **하루 3회 자동 수집·분석**을 통해 빠르게 변하는 선거 정세를 준실시간으로 반영한다. 또한 과거 판정 이력을 시계열로 누적하여 단절 없는 연속적 판세 추적이 가능하다.

### 3.4 확장 가능한 플랫폼

Strategy + Registry 아키텍처 덕분에 다음과 같은 확장이 설정 변경만으로 가능하다:

| 확장 방향 | 방법 |
|-----------|------|
| 새로운 선거구 추가 | `config.yaml`에 선거구·후보 정보 추가 |
| 수집 매체 추가 | 새 Scraper 구현체 등록 |
| 임베딩 모델 교체 | `embedder.type` 변경 (openai → bge_m3 등) |
| LLM 교체 | `scorer.provider` 변경 (openai → anthropic) |
| Vector DB 교체 | `vectordb.type` 변경 (chroma → qdrant 등) |
| 분석 주기 변경 | `schedule.cron` 수정 |

이러한 유연성은 2026년 재보궐선거뿐 아니라, 향후 다른 선거나 정치 이벤트에도 즉시 적용할 수 있는 **범용 선거 분석 플랫폼**으로의 발전 가능성을 제공한다.

## 6. 성능 개선 가이드

### 6.1 쿼리 품질 점검

query_templates.json의 질의어가 VectorDB 청크와 잘 매칭되는지 확인하려면 `--skip-score`로 LLM 호출 없이 검색 결과만 분석한다.

```bash
cd backend
PYTHONPATH=. python -m rag.pipeline --district pyeongtaek_b --skip-score
```

#### 질의별 score 분포 확인

각 질의마다 아래와 같은 로그가 출력된다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
의미 검색 시작 — query='김용남 평택을 지지율'
  필터: district_id=pyeongtaek_b, candidate=김용남
  파라미터: top_k=5, lookback_days=14
  VectorDB 검색 결과: 5건 (필터 적용)
  [1] score=0.8723 | 김용남 지지율 28%로 선두 | naver_news | 2026-05-08
  [2] score=0.8145 | 평택을 여론조사 결과 발표 | naver_news | 2026-05-07
  [3] score=0.6231 | 평택시 재개발 논란 | naver_news | 2026-05-06
  [4] score=0.4102 | 경기도 교육감 인사 | naver_news | 2026-05-05
  [5] score=0.3011 | 국회 본회의 개원 | naver_news | 2026-05-04
```

#### 점검 기준

| 체크 항목 | 양호 | 문제 |
|----------|------|------|
| 상위 score | 0.8 이상이 2~3건 | 최고 score가 0.5 이하 → 질의어와 청크가 불일치 |
| score 분포 | 상위는 높고 하위는 낮음 (격차 큼) | 0.3~0.4 부근에 몰림 → 관련 기사 부족 또는 질의어 모호 |
| 제목 관련성 | 질의와 제목이 의미적으로 일치 | "김용남 강점" 질의에 무관한 기사만 검색 |
| 필터 fallback | 필터 적용 시 결과 있음 | "필터 검색 결과 0건" 빈번 → 기사 태깅 누락 |

#### 전체 요약 확인

```
선거구 그룹별 검색 완료 — 평택을, 총 220건 (고유 87건)
그룹별 재정렬 완료 — 220건 → 145건 (min_score=0.30, dedup=True)
```

| 체크 항목 | 양호 | 문제 |
|----------|------|------|
| 고유 청크 비율 | 총 220건 중 고유 87건 (적당한 중복) | 고유 10건 → VectorDB에 데이터 부족 |
| min_score 필터 후 생존율 | 70~80% 생존 | 220건 → 20건 → 대부분 무관한 기사 |
| 후보별 균형 | 후보마다 비슷한 수의 청크 | 특정 후보만 청크가 몰림 |

#### 문제 발견 시 조치

| 문제 | 원인 | 조치 |
|------|------|------|
| 특정 질의의 score가 모두 낮음 | 질의어가 기사 표현과 동떨어짐 | `query_templates.json`의 해당 질의어를 기사에서 실제 쓰이는 표현으로 수정 |
| 필터 fallback이 빈번 | 기사 태깅 누락 (`candidate`, `district_id`) | `config.yaml`의 `keywords` 목록 확대 |
| 고유 청크 수가 너무 적음 | 수집된 기사 부족 | `--days` 늘려서 재수집, 또는 `lookback_days` 확대 |
| min_score 필터 후 대량 탈락 | 임계값이 높거나 청크 품질 문제 | `min_score` 낮추거나 chunker 설정 조정 |

### 6.2 검색 파라미터 튜닝

config.yaml에서 공통 쿼리와 후보별 세부 쿼리의 top_k를 차등 적용할 수 있다.

```yaml
rag:
  retriever:
    top_k: 20              # 기본값 (단발 질의, fallback)
    top_k_common: 10       # 공통 쿼리 (판세, 여론조사)
    top_k_candidate: 5     # 후보별 세부 쿼리 (지지율, 공약, 강점...)
    lookback_days: 14      # 최근 N일 기사만 검색
  reranker:
    min_score: 0.3         # 유사도 임계값 미만 제거
    deduplicate: true      # 동일 기사 URL 중복 제거
```

공통 쿼리는 선거구 전반의 판세를 파악하므로 넓게 검색(top_k=10)하고, 후보별 세부 쿼리는 특정 분석 항목에 집중하므로 좁게 검색(top_k=5)하여 관련성을 높인다.

| 설정 조합 | 총 검색량 (평택을 기준) | 특징 |
|-----------|----------------------|------|
| top_k=20 (단일) | 2×20 + 40×20 = 840건 | 넓은 검색, 토큰 초과 위험 |
| common=10, candidate=5 | 2×10 + 40×5 = 220건 | 균형잡힌 설정 (권장) |
| common=5, candidate=3 | 2×5 + 40×3 = 130건 | 최소 검색, 빠른 응답 |

### 6.3 Rerank 단계 — 2단계 재정렬 (구현 완료)

#### bi-encoder의 한계와 Cross-encoder 보완

bi-encoder는 질의와 청크를 **독립적으로 벡터화**한 뒤 코사인 유사도를 계산한다. 빠르지만, 질의와 청크 간의 세밀한 의미 관계(부정, 조건, 맥락)를 놓칠 수 있다. 예를 들어:

| 질의 | 청크 | bi-encoder | Cross-encoder | 실제 관련성 |
|------|------|-----------|--------------|-----------|
| "김용남 강점" | "김용남은 조직력이 탄탄하다" | 0.82 | 0.94 | 높음 (정확) |
| "김용남 강점" | "김용남의 약점은 젊은층 인지도" | 0.71 | 0.23 | 낮음 (bi-encoder 오분류 → Cross-encoder가 보정) |
| "김용남 공약 반응" | "김용남 후보가 공약을 발표했다" | 0.78 | 0.45 | 보통 (발표 사실만, 반응 없음) |

"강점"과 "약점"은 임베딩 공간에서 가까이 위치하므로 bi-encoder만으로는 구분이 어렵다. Cross-encoder는 질의+청크를 하나의 입력으로 결합하여 관련성을 직접 평가하므로 이를 정확하게 구분한다.

#### 구현된 2단계 파이프라인

```
Retrieve (bi-encoder, 빠른 후보 추출)
  → Rerank Stage 1: min_score 필터 + URL 중복 제거 (bi-encoder score 기반)
    → Rerank Stage 2: Cross-encoder 재정렬 (상위 N건만 정밀 평가)
      → Score (LLM 판정)
```

Cross-encoder는 연산이 무거우므로 전체 청크가 아닌 **Stage 1을 통과한 상위 후보**에만 적용하여 비용을 제어한다. 그룹별(후보 × 분석 항목)로 쿼리를 재구성하여 적용한다:
- `_common` 그룹: `"{선거구명} {카테고리}"` (예: "평택을 판세")
- 후보별 그룹: `"{후보명} {카테고리}"` (예: "김용남 지지율")

Cross-encoder의 raw score(logit)는 sigmoid로 정규화되어 0~1 범위의 점수로 변환된다.

#### 설정 (`config.yaml`)

```yaml
rag:
  reranker:
    min_score: 0.3
    deduplicate: true
    cross_encoder:
      enabled: true                    # false면 bi-encoder score만 사용 (기존 방식)
      model: BAAI/bge-reranker-v2-m3   # 한국어 지원 Cross-encoder (~560MB)
      top_n: 10                        # 그룹당 Cross-encoder 통과 후 상위 N건 유지
```

활성화하려면 `sentence-transformers` 패키지 설치가 필요하다:

```bash
cd backend
uv pip install sentence-transformers
# 또는
uv sync --extra reranker
```

최초 실행 시 모델이 HuggingFace에서 자동 다운로드된다 (~560MB). 이후에는 로컬 캐시에서 로딩된다.

**사용 가능한 Cross-encoder 모델**:

| 모델 | 언어 | 크기 | 특징 |
|------|------|------|------|
| `BAAI/bge-reranker-v2-m3` | 다국어 (한국어 포함) | ~560MB | **기본값**, 한국어 성능 우수 |
| `ms-marco-MiniLM-L-6-v2` | 영어 (다국어 가능) | ~80MB | 가장 빠름, 한국어 성능 제한적 |
| `multilingual-e5-large` | 다국어 | ~1.2GB | 높은 정확도, 느림 |

#### min_score 최적화

현재 `min_score: 0.3`이 적정한지 확인하려면 `--skip-score`로 실행하여 score 분포를 분석한다.

```bash
PYTHONPATH=. python -m rag.pipeline --district pyeongtaek_b --skip-score
```

로그에서 확인할 항목:

- **0.3 미만으로 탈락하는 청크 비율**: 50% 이상 탈락하면 임계값이 너무 높거나 데이터 품질 문제
- **0.3~0.5 구간의 청크 품질**: 이 구간의 청크 제목을 직접 확인하여 실제로 무관한 기사인지 판별
- **score 분포의 자연적 경계**: score가 급격히 떨어지는 구간이 있으면 그 값을 임계값으로 설정

| min_score | 효과 | 적합한 상황 |
|-----------|------|-----------|
| 0.2 | 더 많은 청크 통과 → 넓은 근거 | 수집 기사가 적을 때, recall 우선 |
| 0.3 | 현재 기본값 — 균형 | 일반적 상황 (권장) |
| 0.4 | 관련성 높은 청크만 통과 | 수집 기사가 풍부할 때, precision 우선 |

config.yaml에서 즉시 조정 가능하며, CLI에서 `--min-score` 옵션으로 오버라이드할 수 있다:

```bash
PYTHONPATH=. python -m rag.pipeline --district pyeongtaek_b --min-score 0.4 --skip-score
```

### 6.4 데이터 품질 최적화 (구현 완료)

수집 → 태깅 → 청킹 단계에서 VectorDB에 저장되는 데이터의 품질을 높여 검색 정확도를 개선한다.

#### 기사 구조 인식 청킹 (ArticleAwareChunker)

기존 `korean_paragraph` 청커는 기사 구조를 무시하고 문단 크기로만 분할하므로, 본문 중간 청크에서 "이 기사가 어떤 후보/선거구에 관한 것인지" 문맥이 유실된다.

**문제 예시**:
```
[chunk_0] "김용남 후보가 평택을 재보궐 선거에 출마를 선언했다. 공약으로..."  ← 문맥 충분
[chunk_1] "이에 대해 지역 주민들은 긍정적인 반응을 보이고 있다..."        ← 누구? 어디?
[chunk_2] "한편 상대 후보 측은 실현 가능성에 의문을 제기했다..."          ← 어떤 선거?
```

`ArticleAwareChunker`는 기사의 **제목·리드·본문** 구조를 인식하여 모든 청크에 문맥을 유지한다:

| 청크 | 구조 | 내용 |
|------|------|------|
| 첫 번째 | `[제목] {title}\n[리드] {lead}` | 기사 핵심 정보 압축 |
| 이후 | `[제목] {title}\n{본문 세그먼트}` | 제목 prefix로 문맥 유지 |

- 본문은 문단(`\n\n`) 또는 줄바꿈(`\n`) 경계에서 분할하여 의미 단위 보존
- 리드 문단은 첫 문단에서 문장 경계까지 추출 (최대 200자)
- `chunk_size`, `overlap` 설정은 기존 청커와 동일하게 적용

```yaml
# config.yaml
chunker:
  type: article_aware          # korean_paragraph → article_aware 변경
  params:
    chunk_size: 400
    overlap: 50
```

**효과**: "김용남 공약 반응" 쿼리 시, 본문 중간 청크도 제목에 "김용남"이 포함되어 있어 높은 유사도를 반환한다.

#### 기사 전문 수집 (fetch_article_body)

네이버 뉴스 검색 결과 페이지에서는 기사 요약(snippet)만 노출된다. 기존에는 이 요약문(평균 157자)을 본문으로 저장했기 때문에, 청킹 후 VectorDB에 저장되는 정보량이 절대적으로 부족했다.

**문제**: 검색 결과 snippet만 수집 → 기사당 1~2개 청크 → 의미 검색 시 핵심 내용 누락

| 스크레이퍼 | 기존 body | 수집 방식 |
|-----------|-----------|-----------|
| NaverNewsScraper | 검색 결과 snippet (평균 157자) | `body = summary` |
| NaverElectionScraper | `[언론사명] 제목` (30~50자) | `body = f"[{press}] {title}"` |

**해결**: `fetch_article_body()` 함수를 `base.py`에 구현하고, 각 스크레이퍼의 `_enrich_bodies()` 메서드에서 기사 상세 페이지를 방문하여 전문을 수집한다.

```
수집 흐름: 검색 결과 파싱 → 기사 목록 확보 → _enrich_bodies() → URL별 상세 페이지 접근 → 전문 추출
```

`fetch_article_body()`의 전문 추출 전략:

1. **다중 CSS 셀렉터 순차 시도** (11개): `#dic_area`, `#articleBodyContents`, `.article_body` 등 주요 매체의 본문 영역 셀렉터를 우선순위 순으로 탐색. 80자 이상 텍스트가 추출되면 즉시 반환.
2. **불필요 요소 제거**: `script`, `style`, `iframe`, `.reporter_area`, `.copyright` 등을 `decompose()`로 DOM에서 물리 삭제 후 텍스트 추출.
3. **og:description Fallback**: 모든 셀렉터 실패 시 `<meta property="og:description">` 태그에서 50자 이상 내용 추출.
4. **안전한 실패**: 네트워크 오류나 HTTP 400+ 응답 시 빈 문자열 반환 (예외 전파 없음).

**효과**: 기사당 평균 본문 길이 157자 → 1,200자 이상으로 증가. 50개 기사에서 54개 청크 → 346개 청크로 수집량 6배 이상 증가.

#### VectorDB 품질 필터 (_filter_chunks)

청킹·태깅 후에도 VectorDB에 노이즈가 유입되는 두 가지 경로를 차단한다:

| 노이즈 유형 | 원인 | 비율 (필터 도입 전) |
|------------|------|-------------------|
| 극소 청크 | 제목만 남은 청크, 리드 1문장 잔여 | 24% (50자 미만) |
| 미태깅 청크 | 선거구와 무관한 일반 뉴스 | 24% (district_id 없음) |

`_filter_chunks()` 조건:
- `char_count >= 50` — 50자 미만 청크 제거
- `district_id` 존재 — 선거구 태깅이 안 된 청크 제거

**효과**: VectorDB에 저장되는 청크가 모두 50자 이상 + 선거구 태깅 완료 상태. 검색 시 무관한 결과가 줄어들어 RAG 판정 정확도 향상.

#### 태거 문맥 검증 — 동음이의어 오태깅 방지

**문제**: "조국"이라는 키워드는 정치인 이름이자 "motherland"를 뜻하는 일반 명사이기도 하다. 키워드 매칭만으로는 선거와 무관한 기사가 오태깅되어 VectorDB 검색 결과에 노이즈를 유발한다.

| 기사 제목 | 키워드 매칭 | 실제 관련성 |
|-----------|-----------|-----------|
| "조국 후보가 평택을 재보궐에 출마" | "조국" 매칭 | 관련 있음 |
| "고려인은 가슴속에 늘 조국을 품고 살아온 핏줄" | "조국" 매칭 | **관련 없음** (오태깅) |

**해결**: 키워드 매칭 시 **문맥 검증(Context Validation)**을 수행한다. 매칭된 키워드 위치에서 ±50자 윈도우 내에 **선거 맥락 단어**가 존재해야 유효한 매칭으로 인정한다.

```
선거 맥락 단어 (38개):
  후보, 대표, 의원, 출마, 선거, 재보궐, 보궐, 지지율, 공약, 판세, 당선, 득표, ...
  국민의힘, 더불어민주당, 조국혁신당, 진보당, 자유와혁신, 무소속, ...
  국회, 정치, 보수, 진보, 야당, 여당, 지역구, 표심, 민심
```

| 텍스트 | ±50자 윈도우 내 맥락 단어 | 결과 |
|--------|------------------------|------|
| "조국 **후보**가 출마를 선언했다" | "후보" 존재 | 태깅 |
| "가슴속에 늘 조국을 품고 살아온 핏줄" | 없음 | **미태깅** |
| "**조국혁신당**에서 활동하는 조국 대표" | "조국혁신당" 존재 | 태깅 |

추가로 config.yaml에 구체적 키워드("조국 후보", "조국 대표", "조국혁신당")를 확장하여, 문맥 검증 없이도 정확하게 매칭되는 경로를 확보했다.

### 6.5 임베딩 모델 선택 가이드

3종의 임베더를 `config.yaml`의 `embedder.type` 변경만으로 전환할 수 있다.

| 관점 | OpenAI (text-embedding-3-small) | BGE-M3 | KoSimCSE |
|------|------|--------|----------|
| **한국어 성능** | 좋음 (다국어 학습) | 좋음 (다국어 특화) | **가장 좋음** (한국어 전용) |
| **차원** | 1536 | 1024 | 768 |
| **실행 환경** | API 호출 (네트워크 필요) | 로컬 GPU/CPU (~2GB) | 로컬 GPU/CPU (~500MB) |
| **속도** | 빠름 (API 배치) | 느림 (로컬 추론) | 보통 (경량 모델) |
| **비용** | 유료 (토큰당 과금) | 무료 (로컬) | 무료 (로컬) |
| **의존성** | 네트워크 + API 키 | GPU 권장 | GPU 권장 |

#### 이 프로젝트에서 중요한 관점

1. **한국어 의미 유사도 정확도** — "김용남 지지율"로 검색했을 때 실제 관련 청크가 상위에 오는지. 한국어 고유명사(후보명, 지역명)와 선거 용어("판세", "접전", "공천")의 임베딩 품질이 핵심. 이론적으로는 KoSimCSE가 한국어 전용이라 유리하지만, OpenAI도 한국어 성능이 충분히 좋다.

2. **운영 비용 vs 편의성** — 하루 3회 수집 × 100건 기사 × 평균 2~3청크 = 일 600~900건 임베딩. OpenAI text-embedding-3-small 기준 월 $1 미만 수준으로 비용이 매우 저렴하다. 반면 로컬 모델은 GPU 없이 CPU에서 돌리면 속도가 크게 떨어진다.

3. **VectorDB와의 차원 호환** — 현재 ChromaDB는 차원 자동 감지라 문제없지만, Qdrant 등 운영 DB 전환 시 `dimensions` 설정을 맞춰야 한다. 임베더를 바꾸면 **기존 벡터를 전부 재임베딩**해야 한다.

#### 현실적 권장

현재 OpenAI를 사용하는 상태에서, 성능 개선을 위해 임베더를 바꾸는 것보다 **앞서 진행한 청킹 품질 개선(ArticleAwareChunker)과 태거 문맥 검증이 훨씬 효과가 크다**. 임베딩 모델 차이는 검색 품질에 5~10% 영향이지만, 청크에 제목이 있느냐 없느냐는 검색 결과를 근본적으로 바꾼다.

임베더 성능을 실제로 비교하려면, 동일한 청크셋에 대해 3종으로 각각 임베딩 후 동일 쿼리의 검색 결과 상위 10건을 비교하는 **A/B 평가**가 가장 정확하다.

### 6.6 VectorDB 저장 구조

수집 파이프라인의 최종 산출물인 벡터 레코드가 어떤 구조로 저장되는지 정리한다.

#### 저장 단위

기사 1개는 청킹을 거쳐 복수의 청크로 분할되고, 각 청크가 VectorDB에 **1개의 레코드**로 저장된다.

```
기사 1개 (약 1,200자) → 청킹 → 3~5개 청크 → 임베딩 → 3~5개 벡터 레코드
```

#### 레코드 구성 (ChromaDB 기준)

| 필드 | 타입 | 설명 |
|------|------|------|
| **id** | `string` | `sha256(article_url + chunk_index)`의 앞 16자 — 결정적 ID |
| **embedding** | `float[1536]` | OpenAI text-embedding-3-small 벡터 |
| **document** | `string` | 청크 원문 텍스트 |
| **metadata** | `dict` | 검색 필터 + 출처 추적용 메타데이터 |

#### 메타데이터 필드

```json
{
  "text": "[제목] 평택을 재보궐 판세 분석\n본문 내용...",
  "chunk_index": 0,
  "char_count": 303,
  "article_url": "https://n.news.naver.com/article/...",
  "source": "naver_news",
  "title": "평택을 재보궐 판세 분석",
  "published_at": "2026-05-14T09:30:00",
  "published_at_ts": 1778930200.0,
  "candidate": "김용남",
  "district_id": "pyeongtaek_b",
  "chunker_type": "article_aware"
}
```

- `published_at_ts`: ChromaDB에서 숫자 비교(`$lt`, `$gt`)를 위해 ISO 문자열과 별도로 **타임스탬프(float)**를 저장
- `district_id`, `candidate`: 검색 시 필터 조건으로 사용

#### 검색 흐름

```
쿼리 텍스트 ("김용남 지지율")
    ↓ embed_query()
쿼리 벡터 [0.12, -0.34, ...]  (1536차원)
    ↓ cosine similarity 검색
    ↓ filters: { district_id: "pyeongtaek_b", published_at_ts: { $gt: cutoff } }
    ↓
상위 top_k개 결과 반환 (score + metadata + text)
```

필터 조합:
- `district_id` — 특정 선거구 청크만 검색
- `candidate` — 특정 후보 청크만 검색
- `published_at_ts` — 최근 N일 기사만 검색 (`lookback_days`)
- 다중 필터 시 ChromaDB `$and` 래퍼로 결합

#### 안전장치 3중 구조

| 장치 | 방식 | 역할 |
|------|------|------|
| 결정적 ID | `sha256(url + chunk_index)` | 동일 기사 재수집 시 upsert(덮어쓰기)로 중복 방지 |
| 시간 필터 | `lookback_days` | 검색 시 최근 N일만 사용하여 오래된 데이터 자연 배제 |
| 만료 정리 | `delete_older_than(days)` | `published_at_ts < cutoff` 벡터를 물리 삭제 |

#### 물리 저장

개발 환경에서는 ChromaDB `PersistentClient`로 `.chroma/` 디렉토리에 파일 저장. 운영 환경에서는 `config.yaml`의 `vectordb.type`을 `qdrant`로 변경하면 Docker 기반 Qdrant 서버로 전환된다.

```
backend/.chroma/
  └── {collection_uuid}/       ← election_chunks 컬렉션
      ├── data_level0.bin      ← HNSW 인덱스 + 벡터 데이터
      ├── header.bin
      └── length.bin
```
